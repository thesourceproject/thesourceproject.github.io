Basics of MySQL

4 Jul '21 | 25 min read

#backend #database #sql #mysql #beginner

We will learn the basics behind MySQL. After learning these, you will be able to create dynamic web applications. Even if you prefer other SQL databases such as MariaDB or PostgreSQL, this tutorial will be helpful as the difference between them is mostly syntactical.

Getting Started

If you haven't installed MySQL yet, refer to our previous tutorial.

We will cover the following in this tutorial.

  • How to create databases and tables.
  • Data types in SQL.
  • Performing CRUD(create/read/update/delete) operations.

Before diving into SQL, we need to plan our database first which includes the tables along with their respective columns any relations.

We can see in our company database that there are 2 tables named employee and department. As you might have guessed, the head column of department is a foreign key that refers to the primary key of employee. But what is a primary key?

In simple words, a primary key is a column or a group of columns that can uniquely identify a row. Because it can refer to exactly 1 row, it needs to be unique. For example, if all employees are guaranteed to have a different email address, it may be used as a primary key.

However, it is not considered the best practice to use values that describe the data. This is why we will use an integer id which will auto-increment starting from 1, thus ensuring uniqueness. Also, because the number itself contains no data, it will never need to change later.

Enough theory. It's time to code. Let's start by creating our database. 

CREATE DATABASE company;
USE company;

This will create a new database named company and make it active.

Now we need to create our tables and select the appropriate data type for each column. This is similar to selecting data types for variables in programming viz. by considering its possible values.

You can refer to the official website for a complete list of data types. 

CREATE TABLE employee (
id INT NOT NULL AUTO_INCREMENT PRIMARY KEY,
name VARCHAR(50) NOT NULL,
email VARCHAR(50) NOT NULL UNIQUE,
phone VARCHAR(15) NOT NULL,
date_of_birth DATE NOT NULL,
joining DATE NOT NULL DEFAULT (CURRENT_DATE)
);

We have created a new table employee with the following columns.

  • id - an integer value that will auto-increment from 1. This will be our primary key. NOT NULL ensures that a field is required.
  • name - a variable-length string with a maximum length of 50.
  • email - similar to name but with a UNIQUE constraint so that no two employees can share the same email address.
  • phone - it is possible for phone numbers to have special characters like '+'. So they are stored as strings and not integers.
  • date_of_birth - the DATE data type stores dates in 'YYYY‑MM‑DD' format.
  • joining - the DEFAULT constraint sets a default value for this field if it is not given. In this case, the current date.

Now we will create our second table department. But first, let's understand foreign keys.

A foreign key is a column or group of columns that refer to the primary key of another table. This is one of the main benefits of relational databases as it helps in reducing data duplication.

In our case, each department has a head who is an employee. Instead of duplicating all of the employee details in department, we can refer to that employee using a foreign key. Later, if the head of a department changes, we only need to change the foreign key to refer to another employee, thus saving time. 

CREATE TABLE department (
id INT NOT NULL AUTO_INCREMENT PRIMARY KEY,
name VARCHAR(50) NOT NULL UNIQUE,
location VARCHAR(50) NOT NULL,
date_established DATE NOT NULL DEFAULT (CURRENT_DATE),
head INT,
FOREIGN KEY (head) REFERENCES employee(id) ON DELETE SET NULL
);

We have created a new table employee with the following columns.

  • id, name, location, date_established - these are similar to our previous table.
  • head - this will be our foreign key. It has the same data type as the column it is referring to. We've also specified an action to take if its referred row gets deleted. In our case, if the head employee gets deleted from the employee table, the department's head will be set to NULL. This will ensure that there are no references to non-existent employees.

Notice - The order of creation of tables is important. The parent table must be created before the child table. Here, the department table is the child table as it has a foreign key referring to employee.
Also, as a result, circular dependencies cannot exist.

Our tables are ready to be filled with data. Let's start with employee. 

INSERT INTO employee (name, email, phone, date_of_birth) VALUES
("John Smith", "johnsmith@mail.com", "12345", "1985-11-02"),
("Mike Walker", "mikewalker@mail.com", "24613", "1989-02-12"),
("Steve Simpson", "stevesimpson@mail.com", "11472", "1972-05-25"),
("Jack Robinson", "jackrobinson@mail.com", "27465", "1991-09-18");

The stored data can be seen using the SELECT command. You can also retrieve specific columns by mentioning the column names. 

SELECT * FROM employee;
SELECT id, name, email FROM employee;

The id field does not need to be inserted and increments automatically.

We can filter rows using the WHERE clause with a condition. 

SELECT * FROM employee WHERE id > 1;

Now, let's create some departments. In the head column, we will insert the id of the employee that we want to refer to. For example, to make John Smith the head of Research, we will use his employee id i.e. 1. 

INSERT INTO department (name, location, date_established, head) VALUES
("Research", "Portland, Oregon", "2010-07-21", 1),
("Human Resources", "Jacksonville, Florida", "2011-03-02", 3);

Let's look at rows of department.

We can only see the employee id in the head column. So, how can we retrieve the details of the employee that is being referred?

The JOIN clause. This is one of the most powerful features of RDBMS. The JOIN clause compares every row from the first table with every row from the second table based on some condition known as the predicate.

Here, we want to join the two tables and only retrieve the rows where the id value of employee is equal to the head value of department. 

SELECT * FROM department JOIN employee ON department.head = employee.id;
SELECT department.name, location, employee.name, phone FROM department JOIN employee ON department.head = employee.id;

Once again, we can specify the columns we want to retrieve. Keep in mind, when a column name is same in both the joined tables, we also specify the table name.

Rows can be deleted using the DELETE command combined with the WHERE clause. Don't forget to specify the condition, or else you may end up deleting every row of the table. 

DELETE FROM employee WHERE id = 4;

The UPDATE command is used to update rows in a table. The WHERE clause is used to specify the row/rows. Just like DELETE, use this command carefully. 

UPDATE department SET date_established = "2010-09-19" WHERE id = 2;

We've covered the absolute basics of MySQL. There is a lot more to learn about databases, but it's going to be a much easier journey from here on.
Try to experiment with SQL on your own and come up with databases for different real-life use cases.
